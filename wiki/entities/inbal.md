# Inbal Shani

Inbal Shani is Chief Product Officer at GitHub, where she leads core product management, product strategy, marketing, open source, and communities. She previously served as a General Manager at AWS and Microsoft, and was a senior software developer on Amazon Robotics. She is at the forefront of the AI transformation in software development, overseeing GitHub Copilot and the future of the GitHub platform.

## Key Views

### Copilot Is a Copilot, Not a Pilot
Inbal's central message: AI tools augment developers, they don't replace them. "Copilot is a copilot, it's not a pilot. You still need the human in the loop." But this shifts the developer's role from writing code to thinking about systems, architecture, and the big picture. Junior developers benefit enormously — instead of spending years learning syntax, they can focus on understanding the system and product from day one. "The user of the AI tools needs to form a different thinking."^[raw/transcripts/lenny/inbal-s.md]

### Overhyped: AI Will Replace Humans; Underhyped: AI-Driven Testing
Inbal distinguishes what's genuinely new from what's noise. Overhyped: "Generative AI will replace humans. I don't see that happening in the near future. That creative spark, that creative thinking that is the center of humanity, this will not be replaced by AI." Underhyped: AI-driven testing (unit, integration, load, security, penetration). With more code being generated, the importance of automated testing across all dimensions is massively underappreciated. "If we're now saying software is in the center of everything, there is more code being generated — the importance of testing is becoming much more critical."^[raw/transcripts/lenny/inbal-s.md]

### Work Backwards From the Customer Problem, Not From "We Need AI"
Inbal's biggest advice for product teams rushing into AI: don't start with "what should we do with AI?" Start with the customer problem. "A lot of companies expect a change to happen magically. There's often the question of 'What should we do with AI?' because if not, I'm not doing the right thing. Versus the question: what is that problem that we're trying to solve and how can we leverage AI better to help solve the problem?" The AI should be in service of the customer need, not the other way around.^[raw/transcripts/lenny/inbal-s.md]

### Developer Happiness Is the Ultimate Metric
GitHub tracks multiple metrics for Copilot's impact: 55% faster code writing, 85% of participants feeling more confident in code quality, code reviews completed 15% faster, and 88% feeling less frustrated and more focused. But Inbal emphasizes that developer happiness is the ultimate north star. Developers spend less than 20-25% of their time actually writing code — the rest is meetings, reviews, waiting on builds. If AI can give them time back, they're "more happy, more productive, more creative."^[raw/transcripts/lenny/inbal-s.md]

### Seamless Integration Is the Design Philosophy
Copilot's design philosophy is minimal friction. "If it's an extra tool and if you need to ask for it or wait for it, developers will not adopt it." The tool must be seamlessly integrated into the existing workflow — so intuitive that the developer barely notices it's there. GitHub eats its own dogfood: "Nothing is shipped before it spends enough months cooking inside GitHub. If we cannot use them, our customers cannot use them for sure." This ensures every feature is battle-tested by the people building it.^[raw/transcripts/lenny/inbal-s.md]
