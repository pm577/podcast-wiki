# Inbal Shani

**Role:** Chief Product Officer at GitHub; former GM at AWS and Microsoft; former senior software developer on Amazon Robotics  
**Known for:** Leading GitHub Copilot — the most widely adopted AI coding assistant with 1.5M+ developers and 37K+ organizations; expertise in AI-powered developer tooling and product strategy.

## Key Views

### 1. Copilot Is a Copilot, Not a Pilot — Humans Stay in the Loop

Inbal's central thesis: AI will not replace software engineers. "Copilot is a copilot, it is not a pilot. You still need the human in the loop." The role of developers shifts from writing code to thinking about systems, architecture, and the big picture. Junior developers can now spend less time learning syntax and more time understanding the system they're building — effectively accelerating their growth to senior-level thinking. 92% of developers already use AI tools, and Copilot users report coding 55% faster, with 85% feeling more confident in code quality.^[raw/transcripts/lenny/inbal-s.md]

> "The user of the AI tools to develop software needs to form a different thinking. You need to start figuring out how are you using these AI tools to help you be successful. And it's no longer just the actual code writing, it's really evolving your thinking to the big picture." — Inbal Shani

### 2. Underhyped: AI-Driven Testing

While everyone focuses on code generation, Inbal believes **AI-driven testing is underhyped**. As AI generates more code faster, the need for comprehensive testing (unit, integration, load, security, and penetration testing) becomes critical. "If we can leverage AI to generate that testing suite that tests everything... from unit testing to load testing to performance testing" — that's the next big opportunity. Testing is often neglected because it's unseen work, but with more code being generated, its importance grows exponentially.

### 3. Avoid "Plastering AI on Everything" — Work Backwards from Problems

Inbal warns against the common mistake of asking "What should we do with AI?" instead of "What problem are we trying to solve?" Companies rush to adopt AI because it's trendy, leading to wasted effort. Her approach: start with the developer's pain points (too many tasks, not enough coding time, code review friction), then use AI as the tool to solve those specific problems. This "working backwards from the customer problem" is the design philosophy behind every Copilot feature.

### 4. The Design Philosophy: Zero Friction Integration

Copilot's success comes from its seamless integration — it fades into the background. The principle: "If it's an extra tool and if you need to ask for it or wait for it, developers will not adopt it." Developers already have too many tools and tasks; adding more friction kills adoption. The team "ate their own dog food" — nothing shipped before spending months cooking inside GitHub itself. The grounding principle was "the developer needs to want to use this tool."

### 5. Measuring AI Productivity Goes Beyond Time Saved

Inbal challenges simplistic productivity metrics. "Time is not quantifiable as a success metric because you can write really bad code really fast." Instead, she advocates for measuring **time-to-value** — from when a developer starts a task to when they realize its full value (revenue, adoption, time to market). A holistic approach combines: code quality, security improvements (secrets prevented from leaking), developer collaboration, time gain, and ultimately **developer happiness**. The combination of input metrics leading to developer happiness is what GitHub focuses on.
